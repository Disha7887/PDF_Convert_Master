 Add two new conversion tools to the PDF toolkit, following the EXACT same architecture
  as the existing "Document Restore" tool (git commit 3078a71). Do not invent a new
  pattern — mirror how Document Restore is wired across all layers.

  - **Lock PDF**: user uploads a PDF + a password → returns a password-protected
    (AES-256 encrypted) PDF.
  - **Unlock PDF**: user uploads an encrypted PDF + its password → returns a
    decrypted, openable PDF. (PDF only — images have no password to remove.)

  ## Engine (IMPORTANT — pdf-lib cannot encrypt/decrypt)
  The codebase only has `pdf-lib`, which CANNOT password-protect or decrypt PDFs.
  Use **qpdf** for both operations:
  1. Add the `node-qpdf2` npm package to `artifacts/api-server/package.json`.
  2. Add the **`qpdf` system binary** to `replit.nix` so the package can run
     (and to the Railway/nixpacks build if applicable).
  - Lock: qpdf AES-256 encrypt using the user's password as both user & owner password.
  - Unlock: qpdf decrypt using the supplied password; if the password is wrong or
    missing, return a clean error message (not a crash).

  ## New requirement vs. Document Restore: a password input
  Document Restore took no parameters. These tools need a **password field** from the
  user, sent to the backend as `options.password`. You must add a password input to:
  - the web upload flow (the shared `ConversionWorkflow` component), shown only for
    these two tools, and
  - the mobile convert screen (`app/convert/[toolId].tsx`), shown only for these tools.
  Pass the value through to `/api/convert` in the `options` object.
