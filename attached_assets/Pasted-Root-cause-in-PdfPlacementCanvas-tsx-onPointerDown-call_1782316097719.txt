Root cause in PdfPlacementCanvas.tsx:
  - onPointerDown calls (e.target as Element).setPointerCapture(e.pointerId), capturing the pointer on
  the grabbed CHILD element (the move box or the small resize handle).
  - But onPointerMove, onPointerUp, and onPointerLeave are attached to the ANCESTOR wrapper div
  (ref={wrapRef}).
  - With capture on a child, the wrapper fires pointerleave at drag start, which runs endDrag() and sets
  drag.current = null. onPointerMove then hits "if (!d) return" and bails, so nothing moves or resizes.
  - The rich editor src/components/PdfEditor.tsx already does this correctly: it captures on the SAME
  element that handles pointermove. Match that pattern.

  Required changes in PdfPlacementCanvas.tsx (keep the geometry math — getScale, move clamping, resize
  aspect logic — exactly as-is; only fix capture target + the leave handler):
  1. In onPointerDown, capture on the wrapper instead of the child: use
  wrapRef.current?.setPointerCapture(e.pointerId). Keep e.preventDefault() and e.stopPropagation().
  2. Remove onPointerLeave={endDrag} from the wrapper div (it is what prematurely ends the drag). End the
  drag only on onPointerUp and onPointerCancel. With proper capture, pointerup still fires on the
  wrapper even when the pointer leaves the element bounds.

  Also check the web rich editor (src/components/PdfEditor.tsx):
  - onElementPointerDown returns early unless the active tool is "select" (if (tool !== "select")
  return). If a user adds text/an element and then cannot move it, auto-switch the active tool to
  "select" right after an element is added so it is immediately draggable. Confirm move + corner-resize
  work for text, image, and shape elements.

  Web verification:
  - Sign PDF: create a signature, drag the box to move, drag the bottom-right handle to resize — both
  work on desktop mouse AND touch.
  - Add Image PDF: same move + resize behavior works.
  - Rich editor: add text/image/shape, then move and resize each.
